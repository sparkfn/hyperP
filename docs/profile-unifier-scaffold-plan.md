# Profile Unifier Scaffold Plan

## Technology Stack

| Concern | Choice | Rationale |
|---------|--------|-----------|
| Runtime | Node 20 LTS | Stable, long-term support |
| Language | TypeScript 5.7+ | Type safety across the full backend |
| Framework | Fastify 5 | Fastest Node framework, plugin architecture, JSON Schema validation |
| ORM | Prisma 6 | Type-safe schema mapping to all 18 tables, built-in migration tooling |
| Validation | Zod | Runtime validation for API contracts |
| Queue | BullMQ + Redis | Background jobs: ingestion, identifier aging, retention cleanup |
| Auth | @fastify/jwt + RBAC middleware | 6 roles: ingest_service, read_service, service, support_agent, reviewer, admin |
| Testing | Vitest | Fast, TS-native |
| Fuzzy matching | fuzzball | Jaro-Winkler, Levenshtein for candidate scoring |
| Phone normalization | libphonenumber-js | E.164 normalization |
| LLM (Phase 5+) | @anthropic-ai/sdk | Claude API for structured adjudication |
| Logging | pino (Fastify default) | Structured JSON logs with request IDs |
| Linting | Biome | Replaces ESLint + Prettier in one tool |

## Scaffold Steps

### Step 1: Project Init

Target: ~30 minutes

- Initialize npm project with TypeScript
- Install core dependencies (Fastify, Prisma, BullMQ, Zod, pino)
- Configure TypeScript (strict mode, path aliases)
- Configure Biome (linting + formatting)
- Create docker-compose.yml with PostgreSQL 16 + Redis 7
- Bootstrap Fastify server with pino logging, request-id, error handler
- Add health check endpoint (`GET /v1/health`)
- Add npm scripts: `dev`, `build`, `start`, `test`, `lint`

### Step 2: Prisma Schema

Target: ~2-3 hours

- Translate all 18 tables from profile-unifier-sql-schema.md into schema.prisma
- Define all 12 enum types
- Add check constraints via Prisma where supported, otherwise raw SQL in migration
- Add indexes (partial unique indexes via raw SQL)
- Run initial migration
- Verify generated SQL against the reference schema

### Step 3: Common Middleware

Target: ~1-2 hours

- JWT authentication plugin for Fastify
- RBAC guard decorator with role checks
- Idempotency-Key middleware for write endpoints
- Standard response envelope helpers (`{ data, meta }` and paginated variant)
- Shared Zod schemas for enums, pagination, common request/response types
- Standard error response format

### Step 4: Source System Module

Target: ~1 hour

- CRUD routes for source_system
- CRUD routes for source_field_trust
- Seed script for initial source systems and trust settings
- Unit tests

### Step 5: Ingestion Module

Target: ~1 week (aligns with Roadmap Phase 1)

- `POST /v1/ingest/:source` route with idempotency
- Normalizer library: phone (E.164), email, name, DOB, address
- Quality flag detection (placeholders, invalid format, null-like values)
- Idempotent source record persistence via record_hash
- Source record rejection capture for normalization failures
- BullMQ worker for async batch ingestion
- Ingest run tracking
- Tests: normalization edge cases, idempotency, rejection handling

### Step 6: Person + Deterministic Matching

Target: ~1 week (aligns with Roadmap Phase 2)

- Person service: create, lookup by ID, search by identifier
- Candidate generation with blocking keys
- Deterministic rules engine (verified govt ID match, conflicts, manual locks)
- Merge logic with path compression on merged_into_person_id
- Merge lineage append-only string maintenance
- Merge event audit logging
- No-match lock support (person_pair_lock)
- Golden profile initial computation with survivorship rules
- Golden profile lineage tracking
- Person API routes (profile, source records, audit)
- Tests: merge/unmerge correctness, path compression, deterministic rules

### Step 7: Heuristic Matching

Target: ~1 week (aligns with Roadmap Phase 3)

- Feature extraction from candidate pairs
- Weighted scorer with conditional caps (not simple additive)
- Confidence band configuration (>=0.90 auto-merge, 0.60-0.89 review, <0.60 no-match)
- Review case creation for review-band decisions
- Benchmark harness: run labeled pairs, report precision/recall by band
- Integration with candidate generation pipeline
- Tests: scoring edge cases, threshold behavior

### Step 8: Review Operations

Target: ~1 week (aligns with Roadmap Phase 4)

- Review case queue with priority ordering
- Review action routes: merge, reject, defer, escalate, manual_no_match
- Assignment and SLA tracking
- Unmerge flow with post-merge record flagging
- Manual no-match locks that survive reprocessing
- Tests: full review lifecycle, unmerge correctness

### Step 9: Events and Observability

Target: ~3-4 days

- `GET /v1/events?since=` polling endpoint with cursor pagination
- Prometheus metrics via fastify-metrics
- Identifier aging background worker (last_confirmed_at checks)
- Retention cleanup background worker (retention_expires_at)
- Structured log correlation across ingestion pipeline

## Infrastructure

### docker-compose.yml

- PostgreSQL 16 with pgcrypto and citext extensions
- Redis 7 for BullMQ broker and lightweight caching
- Named volume for PostgreSQL data persistence

### Environment Variables

- `DATABASE_URL` — PostgreSQL connection string
- `REDIS_URL` — Redis connection string
- `JWT_SECRET` — signing key for auth tokens
- `PORT` — server port (default 3000)
- `LOG_LEVEL` — pino log level (default info)
- `NODE_ENV` — development, test, production
